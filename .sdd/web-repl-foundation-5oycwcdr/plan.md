# Implementation Plan: Web REPL Foundation Slice

**Date**: 2026-03-15 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `.sdd/web-repl-foundation-5oycwcdr/spec.md`

## Summary

Establish the minimum shared groundwork for all later Hydra web initiative phases: a clearly demarcated workspace boundary separating web from core, a shared contract system for cross-surface data and behavioral alignment (browser ↔ gateway ↔ daemon), an enforceable quality gate for all web initiative contributions, and boundary/contract documentation that serves both human maintainers and automated agents.

## Technical Context

**Language/Version**: TypeScript 5.9.x, ESM-only, targeting ES2024. Node.js ≥ 24.0.0 required (native TypeScript execution, no build step).
**Primary Dependencies**: Zod v4.3.6 (already present — used for runtime schema validation of contracts). No new production dependencies required.
**Storage**: N/A — this slice produces documentation, schemas, configuration, and test infrastructure, not runtime data storage.
**Testing**: Node.js native test runner (`node:test` + `node:assert/strict`). c8 for coverage (80% threshold enforced in CI). Stryker for mutation testing on shared modules.
**Target Platform**: Node.js daemon + future browser runtime. Foundation slice itself is platform-agnostic (schemas, docs, config).
**Project Type**: Library/infrastructure foundation within an existing monolithic TypeScript CLI/daemon project.
**Performance Goals**: N/A for foundation slice — no runtime code paths introduced.
**Constraints**: Zero disruption to existing Hydra core. No reorganization of `lib/`, `test/`, `bin/`, or `scripts/`. All existing tests must continue to pass. Existing ESLint boundary rules must not weaken.
**Scale/Scope**: Single-developer-team project with AI agent contributors. Foundation slice supports ~3–5 later web phases.

## Research Findings

All technical decisions are documented in [research.md](./research.md). Summary of six key decisions:

### Decision 1: Workspace Boundary — npm Workspaces with `apps/` and `packages/`

- **Chosen**: npm workspaces adding `apps/web`, `apps/web-gateway`, `packages/web-contracts` incrementally
- **Rationale**: Matches the authoritative docs layout, keeps existing root structure intact, enables per-package deps/tsconfig
- **Alternatives rejected**: Top-level `web/` (doesn't match docs, misses gateway/contracts separation), `lib/web/` (blurs boundary), separate repo (defeats incremental strategy)

### Decision 2: Shared Contracts — Zod Schemas in `packages/web-contracts/`

- **Chosen**: Dedicated `packages/web-contracts/` workspace package with Zod v4 schemas
- **Rationale**: Zod already present, provides types + runtime validation from single definition, dedicated package enforces boundary architecturally
- **Alternatives rejected**: `lib/hydra-shared/contracts/` (couples web contracts to core), JSON Schema (no types without codegen), TypeScript interfaces only (no runtime validation), Protobuf (requires build step)

### Decision 3: Quality Gate — Extend Existing Pipeline

- **Chosen**: Add workspace-specific ESLint boundary rules + per-package tsconfig + inherit existing coverage/lint/typecheck gates + enforce TDD ordering
- **Rationale**: Mature pipeline already exists; extending it is cheaper and more consistent than building separately
- **Alternatives rejected**: Custom gate script (maintenance burden), CODEOWNERS only (advisory, not enforced), separate CI workflow (risks divergence)

### Decision 4: Boundary Documentation — Structured Markdown + ESLint Enforcement

- **Chosen**: `docs/web-interface/07-boundaries-and-governance.md` (subordinate to the existing `docs/WEB_INTERFACE.md` document set) with YAML front matter + ESLint config as machine-enforced truth
- **Rationale**: Fits within the existing numbered web-interface doc set; authority flows from `docs/WEB_INTERFACE.md` as the index; front matter is agent-parseable; ESLint config is the real enforcement
- **Alternatives rejected**: JSON/YAML config (poor narrative), README.md only (not standalone)

### Decision 5: Contract Versioning — File-Level Version Numbers

- **Chosen**: Explicit version in filenames (`session-contract-v1.ts`) + `CONTRACTS.md` lifecycle index in `packages/web-contracts/`
- **Rationale**: Simplest scheme; new version = new file; zero tooling required; old versions remain until consumers migrate
- **Alternatives rejected**: Semver registry (overengineered), date-based (less informative), edit-in-place (breaks FR-007)

### Decision 6: Extensibility — Convention-Based Extension Points with Append-Only Registries

- **Chosen**: Append-only registries (barrel exports, contract index), additive workspace packages, additive ESLint rules. Structural artifacts (boundary docs, quality gate, workspace config shape) remain untouched.
- **Rationale**: Later phases extend without modifying foundation structural artifacts; append-only registries are designed for growth; npm workspaces make adding packages a well-defined operation
- **Alternatives rejected**: Plugin registry (runtime complexity), monorepo-per-phase (packages by responsibility not by phase), strict zero-touch (unrealistic for barrel exports)

## Data Model

The foundation slice does not introduce runtime data storage. The key entities are schema definitions and documentation artifacts. See [data-model.md](./data-model.md) for the contract entity model.

### Core Entities

| Entity                        | Description                                                                     | Key Attributes                                                                                                               |
| ----------------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **Shared Contract**           | Versioned schema defining cross-surface data shapes and behavioral expectations | name, version, status (draft/stable/deprecated), consumers list, Zod schema, inferred TypeScript type                        |
| **Workspace Boundary**        | Documented separation between web initiative packages and Hydra core            | ownership scope per directory (`apps/`, `packages/`, `lib/`), allowed import directions, governance rules, exception process |
| **Quality Gate**              | Enforceable quality expectations for web initiative changes                     | TDD methodology, testing threshold, validation requirements, documentation requirements, audit trail expectations            |
| **Contract Conformance Test** | Test that validates real data against a published contract schema               | target contract + version, test data fixtures, expected pass/fail outcomes                                                   |

### Relationships

- A **Shared Contract** is consumed by one or more **Surfaces** (browser, gateway, daemon).
- A **Contract Conformance Test** validates data against exactly one **Shared Contract** version.
- The **Quality Gate** applies to all changes within the **Workspace Boundary** (`apps/` and `packages/`).
- The **Workspace Boundary** references the ESLint boundary configuration and per-package tsconfig as enforcement mechanisms.

## Interface Contracts

The foundation slice exposes three categories of interfaces. See [contracts/README.md](./contracts/README.md) for detailed definitions.

### 1. Shared Contract Definitions (programmatic)

Each contract in `packages/web-contracts/src/` exports:

- A Zod schema object (runtime validator)
- An inferred TypeScript type (compile-time safety)
- A `parse()` function that returns validated data or a structured error

**Foundation scope**: The foundation slice creates the package scaffold, contract infrastructure (barrel export, test helpers, contract index), and **initial shared vocabulary** — definitional stubs naming the six core protocol objects (Conversation, Turn, StreamEvent, ApprovalRequest, Artifact, SessionSnapshot) and the five first daemon contract families (conversation messaging; command catalog and execution; council and multi-agent eventing; task live output; config and controlled mutations). These are lightweight type-level definitions establishing shared terminology per `docs/web-interface/06-phases-and-sdd.md` Phase 0 scope. The sixth family from `04-protocol.md` (operational intelligence) is deferred to a later phase. Full field-level schemas are the responsibility of later specs (`web-session-and-auth`, `web-conversation-protocol`).

### 2. Boundary Documentation (human + agent readable)

`docs/web-interface/07-boundaries-and-governance.md` exposes (subordinate to `docs/WEB_INTERFACE.md`, which serves as the index and authority root for the entire web doc set):

- YAML front matter with structured metadata (workspace roots, ownership, phase list)
- Ownership table mapping directories to responsible surfaces (`apps/web` → browser, `apps/web-gateway` → gateway, `packages/web-contracts` → shared, `lib/` → Hydra core)
- Cross-boundary change governance rules
- Exception process for ambiguous ownership

### 3. Quality Gate Specification (human + CI readable)

`packages/web-contracts/QUALITY.md` exposes:

- Enumerated quality expectations (TDD methodology, testing, validation, documentation, audit)
- Each expectation classified as **immediately enforced** (lint, typecheck, format — blocked by existing tooling once packages are wired in) or **required standard** (coverage, test presence, TDD methodology — documented expectations verified by reviewer attestation until CI enforcement is wired)
- Pass/fail criteria for each expectation
- Links to enforcement mechanisms (ESLint rules, CI steps, coverage thresholds)
