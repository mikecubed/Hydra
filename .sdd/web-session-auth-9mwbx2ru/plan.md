# Implementation Plan: Web Session & Authentication

**Date**: 2026-03-15 (revised 2026-03-16) | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `.sdd/web-session-auth-9mwbx2ru/spec.md`

## Summary

This plan delivers secure browser authentication, cookie-based session lifecycle management, browser-safety hardening (CSRF, origin validation, CSP, TLS posture), and audit event recording for the Hydra web REPL. The system introduces a gateway-mediated security boundary between the browser and the Hydra daemon, with cleanly separated authentication, session, and security-hardening concerns. Operators get unambiguous session-state visibility, idle-timeout protection, and a complete audit trail — all within the local/trusted-environment deployment model that Hydra targets.

> **Scope note.** Dangerous-action catalogs, audit-log query UX, and admin
> session management are Phase 4 concerns and intentionally excluded. The
> module architecture is designed so those slices can add an authorization
> module without modifying auth or session modules.

## Technical Context

**Language/Version**: TypeScript 5.9+ (strict, ES2024, NodeNext modules)
**Primary Dependencies**: Zod 4.x (schema validation), Hono (gateway framework — per `docs/web-interface/02-stack-and-monorepo.md`), node:crypto (credential hashing, session token generation)
**Storage**: File-system based (daemon-owned state persistence, consistent with existing Hydra event-sourced state model); in-memory session store backed by periodic file snapshots
**Testing**: node:test + node:assert/strict (native runner); c8 for coverage; Stryker for mutation testing
**Target Platform**: Node.js ≥ 24.0.0, local/LAN browser access (loopback-only by default)
**Project Type**: Library + gateway service (extends existing Hydra monorepo with new workspace packages)
**Performance Goals**: Authentication completes in < 10 seconds (SC-002); session state transitions communicated within 5 seconds (SC-003)
**Constraints**: No external runtime dependencies beyond what Hydra already uses; no database server; gateway must not become a second control plane; TLS required for non-loopback; ESM-only
**Scale/Scope**: Single operator or small trusted team; not multi-tenant SaaS

## Research Findings

See [research.md](./research.md) for full analysis. Key decisions summarized below.

### Decision: Gateway as the Authentication Boundary

- **Chosen**: The gateway (`apps/web-gateway`) owns browser authentication and session management. The daemon remains the authority for orchestration state.
- **Rationale**: `docs/web-interface/03-architecture.md` explicitly requires that "the gateway is not a second control plane" but that it owns "auth, browser sessions, WebSocket termination." Browser credentials must not be the raw daemon control token (per `docs/web-interface/05-security-and-quality.md`). The gateway is the natural trust boundary between the browser and daemon.
- **Alternatives considered**: (a) Daemon-owned browser sessions — rejected because the daemon's `x-ai-orch-token` model is not browser-session-aware and adding browser concerns would violate the "daemon stays authoritative for orchestration, not browser UX" rule. (b) Stateless browser auth — rejected because session lifecycle visibility (FR-005) and server-side revocation (FR-009) require server-tracked state.

### Decision: Concern Separation Strategy

- **Chosen**: Three physically separate modules in this slice (`auth/`, `session/`, `security/`) with distinct service interfaces and no cross-imports between them. They communicate only through shared contract types from `packages/web-contracts/`. A future `authorization/` module (Phase 4) can be added alongside without modifying existing modules.
- **Rationale**: FR-019 requires extensibility without cross-concern rework. SC-008 requires that adding a new auth method, authorization rule, or session capability changes only one concern. Physical separation enforces this at the import level. The `authorization/` directory is reserved for Phase 4; this slice does not implement it.
- **Alternatives considered**: Single unified auth module — rejected because it couples credential verification with session lifecycle and permission checks, making SC-008 unmeasurable.

### Decision: File-Based Persistence for Sessions and Audit

- **Chosen**: In-memory session store with periodic file snapshots; append-only file-based audit log.
- **Rationale**: Hydra targets local/LAN deployment (not cloud-scale). The daemon already uses file-based event-sourced state. No database dependency exists and adding one contradicts the project's minimalism. File-based storage is sufficient for the scale (single operator or small team) and keeps the deployment model simple.
- **Alternatives considered**: (a) SQLite — viable but adds a native dependency. (b) Redis — rejected, requires a separate service. Either could be swapped in later due to the store abstraction.

### Decision: Abstracted Clock for Session Expiry

- **Chosen**: All time-dependent logic uses an injectable clock abstraction rather than direct `Date.now()` calls.
- **Rationale**: Edge case in spec: "session expiry logic must be resilient to reasonable clock drift; the system should fail closed." A clock abstraction makes this testable (inject a fake clock in tests) and allows adding drift detection without modifying session logic. FR-018 (fail closed on undetermined state) is easier to implement when time is a controlled dependency.
- **Alternatives considered**: Direct `Date.now()` — rejected because it makes time-dependent tests fragile and clock-drift resilience untestable.

### Decision: Session State as Explicit Finite State Machine

- **Chosen**: Session state transitions are modeled as an explicit state machine with defined valid transitions, rather than ad-hoc flag checks.
- **Rationale**: FR-005 requires the system to communicate exactly one of six states. The spec's edge cases (daemon restart, concurrent tabs, clock skew) all involve state transitions. An explicit FSM makes invalid transitions impossible by construction and makes every transition auditable (FR-013). `docs/web-interface/05-security-and-quality.md` says "no hidden state machines in components when explicit state modeling is warranted" — this is warranted.
- **Alternatives considered**: Boolean flags (`isExpired`, `isRevoked`, etc.) — rejected because they allow contradictory states and make exhaustive handling difficult.

## Data Model

See [data-model.md](./data-model.md) for full entity definitions. Summary below.

### Entities

| Entity           | Key Attributes                                                                                | Relationships                                     |
| ---------------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| **Operator**     | id, displayName, createdAt                                                                    | Has 0..N Sessions, 0..N Credentials               |
| **Credential**   | operatorId, type, hashedSecret, createdAt, lastUsedAt                                         | Belongs to 1 Operator                             |
| **Session**      | id, operatorId, state, createdAt, expiresAt, lastActivityAt, extendedCount, csrfToken         | Belongs to 1 Operator; produces 0..N AuditRecords |
| **SessionState** | Enum: `active`, `expiring-soon`, `expired`, `invalidated`, `logged-out`, `daemon-unreachable` | Current state of a Session                        |
| **AuditRecord**  | id, timestamp, eventType, operatorId, sessionId, detail, outcome                              | References Operator and Session                   |

> **Removed from this slice**: `DangerousActionPolicy` (deferred to Phase 4).

### State Machine: Session Lifecycle

```text
[created] → active
active → expiring-soon  (when remaining time < warning threshold)
active → expired         (when time exceeds expiresAt)
active → invalidated     (by system policy: daemon restart, concurrent-session limit)
active → logged-out      (by operator logout)
active → daemon-unreachable (when daemon heartbeat fails)
expiring-soon → active   (when session is extended)
expiring-soon → expired  (when time exceeds expiresAt)
expiring-soon → invalidated (by system policy)
expiring-soon → logged-out (by operator logout)
daemon-unreachable → active (when daemon heartbeat resumes)
daemon-unreachable → expired (when time exceeds expiresAt during outage)
```

Terminal states (no further transitions): `expired`, `invalidated`, `logged-out`.

### Validation Rules

- Session `expiresAt` must always be in the future at creation time
- Session `extendedCount` must not exceed a configurable maximum
- AuditRecord `timestamp` must be monotonically increasing per session
- Operator `id` must be unique across all operators
- Session `id` must be cryptographically random and unguessable
- Credential `hashedSecret` must never be stored in plaintext or reversible encoding

## Interface Contracts

All public interfaces are defined as Zod schemas in `packages/web-contracts/` and shared across browser, gateway, and daemon boundaries.

### Authentication Interface

| Operation    | Input                                                           | Output                                                                                                                                            | Notes                                                                                                                                                                                                                         |
| ------------ | --------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Login        | `LoginRequest { identity, secret }`                             | `LoginResponse { operatorId, expiresAt, state }` + `Set-Cookie: __session=<id>; HttpOnly; SameSite=Strict; Secure` + `Set-Cookie: __csrf=<token>` | Rate-limited (FR-003). Error does not reveal which field failed (FR-002). Session ID is **never** in the response body (FR-020).                                                                                              |
| Logout       | (implicit from session cookie)                                  | `LogoutResponse { success }` + `Set-Cookie: __session=; Max-Age=0`                                                                                | Terminates session on both sides (FR-007).                                                                                                                                                                                    |
| Idle Re-auth | `LoginRequest { identity, secret }` + valid idle session cookie | `SessionInfo { state, expiresAt, operatorId }` (no new cookie)                                                                                    | Requires a valid-but-idle session. Verifies credentials against the session's operator, resets `lastActivityAt`, and resumes the existing session — no new session is created (US-4 scenario 3). Session cookie is unchanged. |

### Session Interface

| Operation              | Input                          | Output                                         | Notes                                                                                                                                                                                                                        |
| ---------------------- | ------------------------------ | ---------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Get State              | (implicit from session cookie) | `SessionInfo { state, expiresAt, operatorId }` | Continuous visibility (FR-005).                                                                                                                                                                                              |
| Extend                 | (implicit from session cookie) | `ExtendResponse { newExpiresAt }` or error     | Policy-gated (FR-008). Requires CSRF token.                                                                                                                                                                                  |
| State Change Broadcast | (server-initiated)             | `SessionEvent { type, newState, reason? }`     | Pushed to all connections for the affected session (FR-016). The gateway routes broadcasts by server-side session ID internally; the event payload sent to the browser does **not** include the session identifier (FR-020). |

### Security Interface (new)

| Concern                       | Mechanism                                                                                  | Notes                                                                                                                                                         |
| ----------------------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CSRF (FR-022)                 | Double-submit: `__csrf` cookie + `X-CSRF-Token` header on mutating routes                  | WebSocket upgrade uses Origin validation instead.                                                                                                             |
| Origin (FR-021)               | `Origin` header must match gateway origin on mutating routes and WS upgrade                | Rejects mismatched or missing Origin.                                                                                                                         |
| Headers (FR-023)              | Hardened headers middleware on every response                                              | CSP, X-Content-Type-Options, X-Frame-Options, HSTS.                                                                                                           |
| TLS (FR-024)                  | Startup guard: gateway binds to loopback by default; non-loopback bind requires TLS config | Loopback-only by default (no TLS required). LAN access is opt-in: binding to a non-loopback address requires cert+key; gateway refuses to start without them. |
| Mutating rate limits (FR-025) | Sliding-window per-source on mutating endpoints and WS session creation                    | Separate from login rate limiter.                                                                                                                             |

### Audit Interface (write side only)

| Operation | Input                                                                 | Output      | Notes                                     |
| --------- | --------------------------------------------------------------------- | ----------- | ----------------------------------------- |
| Record    | `AuditRecord { eventType, operatorId?, sessionId?, detail, outcome }` | (persisted) | Append-only. All 11 in-scope event types. |

> **Deferred**: Audit query routes and filterable review UX (FR-014) are Phase 4.
