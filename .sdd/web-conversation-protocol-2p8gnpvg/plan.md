# Implementation Plan: Web Conversation Protocol Slice

**Date**: 2025-07-14 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `.sdd/web-conversation-protocol-2p8gnpvg/spec.md`

## Summary

Define and implement the conversation protocol layer that enables a browser-native Hydra REPL: persistent conversations with ordered turns, incremental streaming with reconnect-safe resumption, inline approval and follow-up handling, cancellation/retry/fork control flow, artifact association, and structured multi-agent activity representation. This protocol slice is an early dependency for all richer web UI work. The implementation leverages the daemon's existing event-sourcing architecture and extends it with new conversation-aware entities and contract families.

## Prerequisites

This slice depends on prior work and does NOT re-own the following:

- **web-repl-foundation** — establishes npm workspaces (`packages/`, `apps/` directories), the `packages/web-contracts/` package skeleton, and baseline quality gates for new packages. Tasks in this slice assume workspace infrastructure already exists.
- **web-session-and-auth** — owns browser session lifecycle, authentication, operator identity issuance, WebSocket termination, and browser session registry. This slice consumes opaque `operatorId` and `sessionId` values passed through contract inputs. It does not manage their creation, validation, or transport-level session tracking.

## Technical Context

**Language/Version**: TypeScript 5.9+, ESM modules, Node 24+
**Primary Dependencies**: Zod (schema validation), existing Hydra daemon runtime, `@modelcontextprotocol/sdk`
**Storage**: Daemon-side event-sourced persistence (extending `AI_ORCHESTRATOR_EVENTS.ndjson` and snapshot model)
**Testing**: Node.js native test runner (`node:test` + `node:assert/strict`), c8 coverage, Stryker mutation testing
**Target Platform**: Node.js daemon (server-side protocol layer) + shared contract types consumed by future browser and gateway packages
**Project Type**: Library (shared contracts) + daemon extension (protocol implementation)
**Performance Goals**: Stream events delivered to consumers within 1 second of production; reconnect state synchronization completes within 2 seconds for conversations up to 1000 turns
**Constraints**: No new runtime dependencies beyond Zod; daemon must remain backward-compatible with existing CLI and MCP agent workflows; all state changes must be event-sourced; protocol must be transport-agnostic (no WebSocket/SSE/HTTP assumptions baked into the contract layer)
**Scale/Scope**: Single operator, multiple browser sessions; conversations up to thousands of turns; artifact content retrieval is on-demand, not inline

## Research Findings

Full research documented in [research.md](./research.md). Key decisions:

### Decision: Conversation as a New First-Class Entity

- **Chosen**: New Conversation entity, coexisting with existing `ActiveSessionEntry`
- **Rationale**: Conversations have different lifecycle semantics than orchestration sessions — they persist across multiple sessions, support forking, and carry approval state. Conflating them would couple conversation lifecycle to orchestration lifecycle.
- **Alternatives considered**: Thin wrapper around ActiveSessionEntry (rejected: lifecycle coupling); replace ActiveSessionEntry (rejected: too disruptive to existing daemon).

### Decision: Turn Granularity

- **Chosen**: A turn is one operator instruction + all resulting system work. Approvals, agent activity, errors, and cancellation are nested within turns as structured events (StreamEvents, ApprovalRequests, ActivityEntries) — not separate top-level turns.
- **Rationale**: Matches operator mental model. Keeps timeline readable. Multi-agent detail, approval exchanges, and error events are accessible on expansion, not forced into the top-level sequence. Aligns with `04-protocol.md` definition: "one user input and the resulting assistant/system/agent activity."
- **Alternatives considered**: Flat turns per agent action (rejected: unreadable timelines); separate activity log (rejected: loses inline context); approval/error/cancellation as separate turns (rejected: fragments the instruction-to-outcome relationship).

### Decision: Stream Resumability via Daemon Event Sequences

- **Chosen**: Sequence-based resumption using the daemon's existing `EventRecord.seq` numbers.
- **Rationale**: The daemon already has monotonically increasing sequence numbers in its event log. Reusing them avoids inventing a parallel ordering system. Replay from a given `seq` is gap-free at the conversation level on reconnect; per-turn event sequences are strictly ordered but may contain gaps because the daemon interleaves events globally.
- **Alternatives considered**: Timestamp-based (rejected: clock skew); opaque cursor tokens (rejected: unnecessary abstraction).

### Decision: Concurrent Instruction Policy

- **Chosen**: Queue-and-notify. New instructions are queued while work is in progress. Operator sees queue state and can reorder or remove items.
- **Rationale**: Rejecting is frustrating; concurrent execution is complex. Queueing respects operator intent without concurrency hazards.

### Decision: Reference-Based Fork Implementation

- **Chosen**: Forked conversations store a reference to parent + fork-point turn. Pre-fork turns are read from the parent.
- **Rationale**: Avoids duplicating large conversation histories. Preserves lineage. Safe because turns are immutable.

### Decision: Approval Staleness Detection

- **Chosen**: Four-state lifecycle (pending → responded | expired | stale) with context-hash-based staleness detection.
- **Rationale**: Prevents operators from approving actions based on outdated context. Spec explicitly requires this.

### Decision: Windowed Turn Loading

- **Chosen**: Load most recent N turns initially; load earlier history on demand with range-based retrieval.
- **Rationale**: Conversations may grow to thousands of turns. Loading all upfront is impractical.

### Decision: Multi-Tab Conflict Resolution

- **Chosen**: First-write-wins with notification to all sessions.
- **Rationale**: Keeps workflow moving while ensuring convergence. Suitable for single-operator use.

## Data Model

Full entity definitions, attributes, validation rules, and relationships documented in [data-model.md](./data-model.md).

**Key entities**: Conversation, Turn, StreamEvent, ApprovalRequest, Artifact, ActivityEntry, Attribution.

**Core relationships**:

- Conversation 1→\* Turn (ordered by position)
- Conversation 0..1→\* Conversation (fork lineage)
- Turn 1→\* StreamEvent (ordered by sequence)
- Turn 1→\* Artifact
- Turn 1→\* ApprovalRequest
- Turn 1→\* ActivityEntry (multi-agent structure)
- Turn 0..1→ Turn (retry linkage)

**Sequence alignment**: StreamEvent sequences align with the daemon's `EventRecord.seq`, enabling unified event replay across reconnect.

## Interface Contracts

Full contract families documented in [contracts/conversation-protocol.md](./contracts/conversation-protocol.md).

**Seven contract families** (six owned, one dependency):

1. **Conversation Lifecycle** — create, open, resume (reconnect), archive, list
2. **Turn Submission and Streaming** — submit instruction, subscribe to stream, load turn history (windowed)
3. **Approval and Follow-Up** — get pending approvals, respond to approval (with staleness handling)
4. **Work Control** — cancel, retry, fork, manage instruction queue
5. **Artifacts** — list artifacts per turn, get content by id, list artifacts per conversation
6. **Multi-Agent Activity** — get activity entries per turn, filter by agent
7. **Session Synchronization** — ⚠️ **dependency on web-session-and-auth** (browser session registration, heartbeat, and transport-level session tracking are NOT owned by this slice)

**Follow-on protocol families** (out of scope — see `04-protocol.md`):

- Command catalog and execution
- Task live output
- Config and controlled mutations
- Operational intelligence

**Cross-cutting concerns**: Structured errors with context, strict ordering guarantees, idempotent mutations, and daemon-authoritative consistency model.

## Implementation Phases

### Phase 1: Shared Contract Types (packages/web-contracts)

> **Prerequisite**: The `packages/web-contracts/` workspace must exist (established by `web-repl-foundation`).

Define all entity types, contract request/response types, and Zod validation schemas within the existing workspace package. This is the foundation that the daemon, gateway, and browser will all depend on. No runtime behavior — pure types and validation. Tests lead implementation (red-green-refactor).

**Delivers**: FR-001 (conversation/turn structure), FR-003 (stream event types), FR-005 (approval types), FR-011 (artifact types), FR-012 (multi-agent activity types).

### Phase 2: Conversation Persistence — User Story 1 (lib/daemon)

Implement the conversation store as an event-sourced extension of the daemon's existing state management. Conversations, turns, and artifacts are persisted through the existing event log. Snapshot support for fast recovery. Daemon routes for conversation lifecycle. Tests lead implementation per TDD cadence.

**Delivers**: FR-002 (create/continue/retrieve conversations), FR-006 (server-side persistence), FR-010 (fork support), FR-014 (windowed retrieval).

### Phase 3: Stream Lifecycle Management — User Story 2 (lib/daemon)

Implement the stream manager that coordinates turn execution, incremental event delivery, and stream lifecycle (start → progress → complete/fail/cancel). Reconnect resumption from last-acknowledged sequence. Finalized stream content populates the turn's `response` field.

**Delivers**: FR-003 (streaming delivery), FR-004 (stream lifecycle signals), FR-007 (reconnect synchronization), FR-008 (cancellation).

### Phase 4: Approval Flow — User Story 3 (lib/daemon)

Implement the approval request lifecycle including staleness detection, response handling (recorded as `approval-response` StreamEvents within the active turn, not as separate turns), and multi-session conflict resolution.

**Delivers**: FR-005 (inline approvals), FR-015 (multi-session conflict resolution).

### Phases 5–8: Remaining User Stories (P2/P3)

Each user story follows the same TDD cadence: write failing tests → implement to pass → add routes → verify integration. Stories 4–7 (reconnect, cancel/retry/fork, artifacts, multi-agent activity) can execute in parallel after Phase 3 completes.

**Delivers**: FR-006–FR-012, SC-003–SC-008.

### Phase 9: Multi-Session Consistency and Polish

Cross-cutting conflict resolution, documentation updates, quality gate verification, and final success criteria validation.

**Delivers**: FR-015, SC-001 through SC-009 verification.

## Risk Mitigation

| Risk                                         | Mitigation                                                                                                                                             |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Event log growth from streaming turns        | Snapshot compaction (already exists in daemon config: every 500 events). Stream events are consolidated into finalized turn content on completion.     |
| Fork reference chains becoming deep          | Limit fork depth or materialize turns beyond a threshold. Monitor in Phase 2 tests.                                                                    |
| Approval staleness detection adding latency  | Context hash comparison is lightweight. Only performed on view/respond, not continuously.                                                              |
| Backward compatibility with existing daemon  | All new functionality is additive. Existing routes, state shape, and MCP tools are unchanged. New conversation routes are namespaced separately.       |
| Contract package versioning                  | Start at 0.1.0. Treat all types as unstable until the gateway and browser packages are also under development.                                         |
| Foundation slice not yet complete            | This slice cannot begin implementation until `web-repl-foundation` establishes the workspace infrastructure. Tasks declare this dependency explicitly. |
| Session/auth primitives not yet available    | This slice uses opaque identifiers and declares `web-session-and-auth` as a dependency. Integration tests use test-only session stubs.                 |
| Follow-on protocol families not yet designed | This slice provides extensible StreamEvent kinds and contract patterns. Follow-on families extend the existing event stream without breaking changes.  |
